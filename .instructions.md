# 🎯 專案開發準則與擴充規範

本文件定義專案的架構設計哲學與擴充規則，旨在確保系統在持續成長中仍能保持一致性。

---

## 🏗️ 核心架構分層準則

當你需要新增功能時，請根據以下準則決定代碼存放位置：

### 1. `core/` - 系統基礎設施
- **判定準則**：如果這項功能是「低階工具」、「數據存儲」或「與特定 AI 決策無關的服務」，請放在這裡。
- **擴充規範**：
    - 新增基礎服務：建立 `core/<service_name>/` 並包含專屬 `.instructions.md`。
    - 新增工具技能：放在 `core/tools/` 下，必須在 `__init__.py` 註冊。
- **禁忌**：嚴禁包含任何 Agent 的決策邏輯、提示詞或對話狀態。

### 2. `logic/` - 代理人決策與流程
- **判定準則**：如果這項功能涉及「如何思考」、「工作流編排」或「提示詞管理」，請放在這裡。
- **擴充規範**：
    - 新增 Agent：在 `logic/agents.py` 定義節點與工具綁定。
    - 調整工作流：修改 `logic/workflow.py` 的狀態機。
- **禁忌**：嚴禁包含底層 API 調用、數據庫寫入或爬蟲細節。

---

## 🔧 系統擴充標準流程

每當新增功能或檔案時，請遵循以下步驟：

1. **職責定位**：根據「分層準則」確認檔案路徑。
2. **命名規範**：
    - 類別/用途導向命名（例如：`search_tool.py` 優於 `my_tool.py`）。
    - 跨模組調用時，必須在該模組的 `__init__.py` 進行導出。
3. **導入路徑**：一律使用絕對路徑（如 `from core.config import settings`）。
4. **文檔同步**：
    - 若新增的是「基礎設施」或「新模組」，需在其目錄下建立 `.instructions.md`。
    - 若只是在現有模組新增檔案，確保該模組的 `__init__.py` 已更新，**無需更新說明文件本身**（除非改變了擴充規則）。

---

## 📋 共通編碼規範

- **類型提示**：所有公開函數必須標註類型簽名。
- **錯誤處理**：一律使用 `try-except` + `logger` 捕獲，並返回友善的字串給 Agent。
- **日誌標準**：使用繁體中文 + Emoji（✅, ❌, ⚠️）標註操作結果。
- **Google Style Docstrings**：所有函數必須描述 Args, Returns 與用途。

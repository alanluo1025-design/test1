# 🧠 core/rag/ - 向量搜索與 RAG 規範

負責知識庫的持久化、向量化與檢索排序邏輯。

---

## 🏗️ 實作準則

1. **資源單例化**：ChromaDB Client 與模型實例必須在 `service.py` 中實現為單例，避免資源浪費。
2. **封裝技術細節**：對外只暴露業務導向的函數，隱藏向量維度、Collection 名稱等底層細節。
3. **語義切割**：文本切塊邏輯應以「資訊完整性」為優先，而非固定字數切割。

---

## 📦 ChromaDB Collection 命名規範

系統共有三個 Collection，命名與用途固定，不得隨意新增或更名：

| Collection 名稱 | 用途 | Metadata 必要欄位 |
|:---|:---|:---|
| `tech_articles` | Agent 的通用知識庫（爬蟲、PDF 等來源） | `source`（來源網址或檔名） |
| `user_preferences` | 使用者偏好向量（標籤 Embedding） | `user_id`, `tags`, `mapping` |
| `article_vectors` | 使用者典藏文章向量 | `user_id`, `article_id`, `collection_id` |

新增 Collection 前，需先更新此規範並確認無法用現有 Collection + metadata 過濾實現需求。

---

## 🔧 擴展規範

- **功能擴展**：若需新增檢索策略，在 `service.py` 中新增函數並在 `__init__.py` 導出。
- **模型更換**：所有模型名稱透過 `settings.embedding_model_name` / `settings.rerank_model_name` 管理，禁止硬編碼。

---

## ⚠️ 禁忌

- ❌ 不得在 `service.py` 以外的地方直接操作 `chromadb.Client`
- ❌ 不得在 `core/rag/` 以外的地方直接讀寫向量資料庫
- ❌ Collection 名稱不得散落在各工具檔案中，一律由 `service.py` 集中管理

# 資料庫結構設計

> 根據「配套措施」文件整理

---

## 整體儲存架構概覽

| 儲存系統 | 負責範疇 |
|---------|---------|
| **SQLite / PostgreSQL** | 使用者、偏好、新聞、摘要、音檔、留言、典藏、通知 |
| **Redis** | 新聞快取、播放進度、TTS 配額、通知任務佇列 |
| **ChromaDB** | 使用者偏好向量、典藏文章向量 |

---

## 一、關聯式資料庫（SQLite / PostgreSQL）

### 1. `users` 使用者表

```sql
CREATE TABLE users (
  id              VARCHAR(12) PRIMARY KEY,  -- 格式：U2025xxxx 或 M2025xxxx
  name            VARCHAR(100) NOT NULL,
  email           VARCHAR(255) NOT NULL UNIQUE,
  provider        VARCHAR(20) NOT NULL,     -- 'google' | 'discord' | 'facebook' | 'line'
  provider_id     VARCHAR(255) NOT NULL,    -- 第三方平台給的唯一 ID
  role            VARCHAR(10) DEFAULT 'user', -- 'user' | 'admin'
  has_topics      BOOLEAN DEFAULT false,    -- 是否已完成主題選擇（新舊用戶判斷）
  created_at      DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at      DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  UNIQUE (provider, provider_id)            -- 同一平台的 ID 不重複
);
```

**說明：**
- `id` 由後端邏輯生成，格式 `U` + 年份 + 4碼流水號（管理者為 `M`）
- `has_topics` 是新舊用戶判斷的關鍵欄位，Auth.js 登入後檢查此欄位決定是否跳轉主題選擇頁

### 2. `user_notification_prefs` 使用者通知偏好表

```sql
CREATE TABLE user_notification_prefs (
  id               INTEGER PRIMARY KEY AUTOINCREMENT,
  user_id          VARCHAR(12) NOT NULL UNIQUE,
  -- 各管道開關（預設只開 email）
  notify_email     BOOLEAN DEFAULT true,
  notify_discord   BOOLEAN DEFAULT false,
  notify_messenger BOOLEAN DEFAULT false,
  notify_line      BOOLEAN DEFAULT false,
  -- 各管道的目標 ID（對應管道開啟後才需填入）
  email_address    VARCHAR(255),            -- 可覆寫 users.email
  discord_user_id  VARCHAR(100),            -- Discord 用戶 ID（用於 Bot DM）
  messenger_psid   VARCHAR(100),            -- Facebook Messenger PSID
  line_user_id     VARCHAR(100),            -- LINE 用戶 ID
  created_at       DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at       DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (user_id) REFERENCES users(id)
);
```

---

### 3. `preferences` 使用者主題偏好表

```sql
CREATE TABLE preferences (
  id              INTEGER PRIMARY KEY AUTOINCREMENT,
  user_id         VARCHAR(12) NOT NULL UNIQUE,
  tags            JSON NOT NULL,            -- ["大型語言模型", "電腦視覺", ...]，最多15類
  created_at      DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at      DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (user_id) REFERENCES users(id)
);
```

**說明：**
- `tags` 存中文標籤名稱，後端有「標籤對應字典」將其轉為英文技術關鍵字（如 LLM、GPT、Transformer）供 Tavily 搜尋使用
- 寫入後同步觸發 Embedding → 存入 ChromaDB

---

### 4. `news_articles` 新聞文章表

```sql
CREATE TABLE news_articles (
  article_id      VARCHAR(36) PRIMARY KEY,  -- UUID
  source_url      VARCHAR(2048) NOT NULL UNIQUE,
  title           VARCHAR(500) NOT NULL,
  content         TEXT,
  summary         TEXT,
  published_at    DATETIME,
  fetched_at      DATETIME NOT NULL,
  cached_until    DATETIME NOT NULL,        -- 通常為 fetched_at + 24 小時
  link_valid      BOOLEAN DEFAULT true,
  categories      JSON,                     -- ["語言模型", "電腦視覺"]
  created_at      DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at      DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX idx_news_cached_until ON news_articles(cached_until);
CREATE INDEX idx_news_link_valid   ON news_articles(link_valid);
CREATE INDEX idx_news_published_at ON news_articles(published_at DESC);
```

**說明：**
- `source_url UNIQUE` 是增量爬蟲防止重複入庫的核心約束
- 每日 06:00、12:00、18:00 定時爬蟲，每晚掃描 `link_valid` 狀態

---

### 5. `article_summaries` 新聞摘要表

```sql
CREATE TABLE article_summaries (
  summary_id          VARCHAR(36) PRIMARY KEY,  -- UUID
  article_id          VARCHAR(36) NOT NULL UNIQUE,
  summary_content     TEXT NOT NULL,
  word_count          INT,
  llm_model           VARCHAR(50),              -- 'gpt-4', 'gemini-pro' 等
  generation_time_ms  INT,
  cached_until        DATETIME NOT NULL,        -- 生成後 +30 天
  created_at          DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at          DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (article_id) REFERENCES news_articles(article_id)
);

CREATE INDEX idx_summary_cached_until ON article_summaries(cached_until);
```

---

### 6. `audio_caches` 語音檔快取表

```sql
CREATE TABLE audio_caches (
  audio_id          VARCHAR(36) PRIMARY KEY,    -- UUID
  article_id        VARCHAR(36) NOT NULL UNIQUE,
  file_url          VARCHAR(1024) NOT NULL,     -- S3 或本地靜態儲存的 URL
  duration_seconds  INT,
  tts_engine        VARCHAR(50),                -- 'edge-tts' | 'google-cloud' | 'openai'
  file_size_bytes   INT,
  cached_until      DATETIME NOT NULL,          -- 生成後 +30 天
  created_at        DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (article_id) REFERENCES news_articles(article_id)
);

CREATE INDEX idx_audio_article_id   ON audio_caches(article_id);
CREATE INDEX idx_audio_cached_until ON audio_caches(cached_until);
```

**說明：**
- `article_id UNIQUE` 確保每篇文章只生成一個音檔，避免重複 TTS 費用
- 實體音檔（`.mp3`）上傳 AWS S3，DB 僅存 `file_url`
- 每日定時任務清理 `cached_until < NOW()` 的記錄與實體檔案

### 7. `comments` 討論留言表（樹狀結構）

```sql
CREATE TABLE comments (
  comment_id          VARCHAR(36) PRIMARY KEY,   -- UUID
  article_id          VARCHAR(36) NOT NULL,
  user_id             VARCHAR(12) NOT NULL,
  parent_comment_id   VARCHAR(36),               -- NULL 表示頂層留言，有值表示回覆
  content             TEXT NOT NULL,
  is_moderated        BOOLEAN DEFAULT false,      -- true = 已被 AI 審核攔截
  moderation_score    FLOAT,                      -- OpenAI Moderation API 的毒性分數
  created_at          DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at          DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (article_id)        REFERENCES news_articles(article_id),
  FOREIGN KEY (user_id)           REFERENCES users(id),
  FOREIGN KEY (parent_comment_id) REFERENCES comments(comment_id)
);

CREATE INDEX idx_comments_article_id ON comments(article_id);
CREATE INDEX idx_comments_parent_id  ON comments(parent_comment_id);
CREATE INDEX idx_comments_user_id    ON comments(user_id);
```

**說明：**
- `parent_comment_id` 自我關聯實現無限層級巢狀回覆（Adjacency List 模式）
- 使用 SQLite 的 `WITH RECURSIVE` CTE 語法遍歷整棵留言樹
- 留言送出前先呼叫 OpenAI Moderation API，`moderation_score` 超標則攔截不寫入

### 8. `notifications` 通知記錄表

```sql
CREATE TABLE notifications (
  notification_id  VARCHAR(36) PRIMARY KEY,     -- UUID
  user_id          VARCHAR(12) NOT NULL,
  type             VARCHAR(20) NOT NULL,         -- 'reply' | 'summary_done' | 'bookmark_remind'
  channel          VARCHAR(15) NOT NULL,         -- 'email' | 'discord' | 'messenger' | 'line'
  title            VARCHAR(255),
  content          TEXT,
  status           VARCHAR(10) DEFAULT 'pending', -- 'pending' | 'sent' | 'failed'
  retry_count      INT DEFAULT 0,
  sent_at          DATETIME,
  created_at       DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (user_id) REFERENCES users(id)
);

CREATE INDEX idx_notifications_user_id ON notifications(user_id);
CREATE INDEX idx_notifications_status  ON notifications(status);
```

### 9. `collections` 典藏表

```sql
CREATE TABLE collections (
  collection_id    VARCHAR(36) PRIMARY KEY,     -- UUID
  user_id          VARCHAR(12) NOT NULL,
  article_id       VARCHAR(36) NOT NULL,
  ai_summary       TEXT,                        -- LLM 改寫的摘要
  keywords         JSON,                        -- AI 擷取的核心關鍵字 ["Transformer", ...]
  vector_id        VARCHAR(100),                -- 對應到 ChromaDB 的向量 ID
  collection_date  DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  UNIQUE (user_id, article_id),                -- 同一篇文章不能收藏兩次
  FOREIGN KEY (user_id)    REFERENCES users(id),
  FOREIGN KEY (article_id) REFERENCES news_articles(article_id)
);

CREATE INDEX idx_collections_user_id         ON collections(user_id);
CREATE INDEX idx_collections_collection_date ON collections(collection_date);
```

**說明：**
- 寫入前執行 `SELECT COUNT(*)` 檢查上限，達到 50 則時先刪除 `collection_date` 最舊的一筆
- `UNIQUE(user_id, article_id)` 在資料庫層防止重複收藏
- 每日凌晨 APScheduler 執行 `cleanup_job()`，刪除超過 30 天的典藏資料

## 二、Redis 資料結構

| Key 格式 | 類型 | 說明 | TTL |
|---------|------|------|-----|
| `news:topic:{topic}` | Sorted Set | 各主題新聞的 `article_id`，Score = 發佈時間戳 | 無（用 `ZREMRANGEBYRANK` 限制最新 1000 筆） |
| `news:article:{article_id}` | Hash | 新聞元數據（title, url, summary...） | 24 小時 |
| `user:{user_id}:playback` | Hash | 各文章播放進度（秒數） | 無 |
| `user:{user_id}:tts_chars` | String | 當月已用 TTS 字數 | 至月底 |
| `notify:queue` | List | Celery 通知任務佇列 | — |

**動態聯集聚合流程：**
1. 取得該次請求的主題列表
2. `ZUNIONSTORE` 合併多主題 Sorted Set 為臨時 Set（TTL: 60 秒）
3. `ZREVRANGE` 對臨時 Set 分頁取出 `article_id`
4. Pipeline 批量向 Hash 取回文章詳細內容

## 三、ChromaDB 向量資料庫

### Collection 1：`user_preferences`（使用者偏好向量）

| 欄位 | 說明 |
|------|------|
| `user_id` | 使用者 ID |
| `tags` | 原始中文標籤列表 |
| `mapping` | 對應英文技術關鍵字列表 |

- **用途**：數位雙生 Agent 比對「使用者已知知識範疇」，過濾重複舊知
- **向量來源**：將 `tags` 文字丟給 Embedding 模型轉換後寫入

### Collection 2：`article_vectors`（典藏文章向量）

| 欄位 | 說明 |
|------|------|
| `user_id` | 收藏者 ID |
| `article_id` | 新聞文章 ID |
| `collection_id` | 典藏記錄 ID |

- **用途**：RAG 查詢時用 `similarity_search` 從使用者 50 則典藏中找出最相關內容餵給 LLM
- **向量來源**：典藏時對 AI 摘要執行 `bge_m3_encode`

## 四、資料表關聯總覽

```
users
  ├── user_notification_prefs  (1:1)
  ├── preferences              (1:1)
  ├── comments                 (1:N)
  ├── notifications            (1:N)
  └── collections              (1:N，上限 50 筆)

news_articles
  ├── article_summaries        (1:1)
  ├── audio_caches             (1:1)
  └── comments                 (1:N)

comments
  └── comments                 (自我關聯，parent_comment_id，樹狀巢狀)
```

## 五、關鍵設計決策說明

| 決策 | 原因 |
|------|------|
| `users.has_topics` 單一布林欄位判斷新舊用戶 | 比每次 JOIN preferences 查詢更快 |
| `comments` 用 Adjacency List（parent_id）而非 Nested Set | 寫入簡單，SQLite RECURSIVE CTE 可處理查詢 |
| `collections` 加 `UNIQUE(user_id, article_id)` | 資料庫層防止重複收藏，不依賴應用層檢查 |
| `audio_caches.article_id UNIQUE` | 每篇文章只生成一個音檔，避免重複 TTS 費用 |
| `news_articles.source_url UNIQUE` | 增量爬蟲防重複入庫的核心約束 |

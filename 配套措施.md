# 配套措施

## 第三方登入配套

### 推薦使用的工具

- **Auth.js**：這是目前開發者社群最推崇的工具，建置在javascript環境裡。它就像一個「萬用插頭」，已經幫你寫好了連接 Google、Discord 等平台的複雜程式碼，會幫忙連接第三方平台，他會讀取env.檔的(client id、client secret、NEXTAUTH_SECRET)
- **SQLAlchemy**：Python 生態的 ORM 工具，讓你用 Python 物件操作資料庫，不必寫複雜的 SQL 指令。本專案以 `core/db/models.py` 定義所有資料表，透過 `get_db()` 依賴注入取得 Session。

### 第三方登入-實現步驟

#### 1.取得開發者金鑰

- 你必須去以下三個平台的開發者後台（Google Cloud Console、Discord Developer Portal、Facebook Developers）註冊你的專案。
- 金鑰產出：你會得到 Client ID（身分證字號）和 Client Secret（私鑰）。
- 建.env檔存起來
```
AUTH_GOOGLE_ID=你的ID
AUTH_GOOGLE_SECRET=你的私鑰
AUTH_DISCORD_ID=你的ID
AUTH_DISCORD_SECRET=你的私鑰
AUTH_FACEBOOK_ID=你的ID
AUTH_FACEBOOK_SECRET=你的私鑰
AUTH_LINE_ID=你的ID
AUTH_LINE_SECRET=你的私鑰
NEXTAUTH_SECRET=固定的隨機字串，可隨機生成(32字元)
```

- 設定 Callback URL：你要告訴這些平台，使用者登入後要跳轉回你的網址(各平台填寫)

#### 2.資料庫表設計

- 一個使用者能多種登入/註冊方式
- User 表：1.ID(資料庫自動生成的ID，一般使用者之格式:U創立年分+4碼流水號、管理者之格式:M創立年分+4碼流水號)、2.姓名 3.Email 4.provider(登入來源)、5.provider_ID(登入來源給的唯一ID)

#### 3.寫入登入邏輯

- 使用者點擊「使用 Google 登入」，後端(Auth.js)會帶著env檔裡的client ID去見第三方平台
- 導向：系統把使用者帶到 Google 的登入頁面。
- 回傳：使用者登入成功後，Google 帶著一個「授權碼」回到你的網站。
- 交換：你的後端拿這個授權碼和client secret去跟 Google 換取使用者的資料（名字、Email、provider、provider_ID）。
- Session (會話)：系統確認這個人是誰，發給他一張「身分證」(Jason web Token)，把JWT鎖進瀏覽器裡的cookie，讓網站記住用戶登入資訊，讓他之後不用重複登入。

### 使用者情況

#### 情況一：註冊 (新使用者)

- 當 Auth.js 換回 Email 後，發現資料庫 User 表 裡查無此人。
- 自動開戶：Auth.js 直接在 User 表新增一筆資料。
- 生成 ID：資料庫自動配發 ID。
- 認定身分：將 ID 鎖進 JWT 通行證。
- 導向功能 2：因為是新朋友，系統判斷他還沒有「喜好主題」，所以畫面會主動跳轉到「請選擇你感興趣的 AI 主題」頁面。

#### 情況二：登入 (已註冊)

- 當 Auth.js 換回 Email 後，發現 User 表 已經有這個人的紀錄了。
- 抓取舊 ID：直接從資料庫抓回他原本的 ID。
- 認定身分：同樣將 ID鎖進 JWT 通行證。
- 導向首頁：系統發現他已經選過主題了，所以不跳轉，直接讓他看到 AI Agent 為他準備好的個人化新聞牆。

## 客製化AI新聞主題

### 客製化AI新聞主題-實現步驟

#### 1.資料設計
- **SQLite 偏好表**：在 `core/db/models.py` 中以 SQLAlchemy 定義 `Preference` 表。欄位：包含 `user_id` 與 `tags`（JSON 陣列，存儲勾選的主題名稱，共15類）。目的：這張表負責「記住」每位使用者選了什麼主題
- **標籤對應字典**：在程式碼中建立一個對照表，管理層需自訂。邏輯：當使用者勾選「大型語言模型」時，後端自動對應到英文技術關鍵字（如 LLM, GPT, Transformer）。目的：讓「抓取 AI 新聞 Agent」有精確的關鍵字去搜尋國內、際新聞。
- **chromaDB metadata表建立**：欄位包含userID、使用者勾選標籤之對應類別(mapping)。目的：讓雙生Agent比對使用者資料，告訴它只看userID= ?的資料，再透過暫存做資料比對，提醒文章略過或是擷取新知。

#### 2.使用者引導與身分識別

當使用者登入後，系統會執行以下流程：

- 登入偵測：透過 Auth.js 取得 JWT，解出 `user_id`。
- 新老用戶判斷：讀取 `users.has_topics` 布林欄位（避免每次 JOIN `preferences` 表，效能更快）。
- 情況 A（`has_topics = false`，新用戶）：強制跳轉至「15 類標籤勾選頁面」。
- 情況 B（`has_topics = true`，老用戶）：直接跳過，進入新聞看板。
- 勾選與存儲：使用者勾選完標籤（例如選了 3 類）並提交。
- 寫入資料庫：將標籤 JSON 陣列存入 `preferences.tags`，同時將 `users.has_topics` 更新為 `true`。

#### 2.1 主題修改流程（老用戶重新選題）

當已設定過主題的用戶想更新偏好時：

- **API 端點**：`PUT /api/user/preferences`，接收新的標籤 JSON 陣列。
- **資料庫更新**：覆寫 `preferences.tags`（`has_topics` 維持 `true`，不重置）。
- **ChromaDB 向量同步**：對 `user_preferences` Collection 執行 upsert——以 `user_id` 為 ID 直接覆蓋舊向量，不需先刪除；`article_vectors` 中的典藏向量不受影響。
- **Agent 即時生效**：下次「抓取 AI 新聞 Agent」執行時，自動以新標籤的對應關鍵字重新搜尋。

#### 3.AI 語意同步與向量化

這一步是為了讓「數位雙生 Agent」能發揮作用

- **語意轉化 (Embedding)**：系統取出剛存入 SQLite 的標籤。將標籤文字丟給 Embedding 模型，轉換成一串數字（向量）。
- **存入 ChromaDB**：將標籤向量存入 ChromaDB。關鍵細節：在 metadata 欄位標記 {"user_id": 1}。目的：建立該使用者的知識樣貌，作為日後比對新舊知識的基準。

#### 4.Agent 鏈鎖反應與結果展示

地基打好後，5 個 Agent 正式開始運作：

- **抓取 AI 新聞 Agent (啟動)**：從 SQLite 讀取標籤，透過「字典」也就是對照表轉化為技術關鍵字。使用 Tavily 搜尋對應的熱門 AI 新聞。
- **數位雙生 Agent (模擬閱讀)**：將抓回來的新聞轉為向量，並在 ChromaDB 中進行比對。若發現新聞內容是使用者已知的（存在暫存），則會提醒略過或僅擷取新知。
- **生成摘要與展示**：通過語言模型將過濾後的新聞濃縮成 100-200 字簡述。最終呈現在「每日新聞看板」介面上供使用者觀看。

## 每日 AI 新聞看板

| 配套項目 | 實現細節 |
|---------|---------|
| 新聞聚合與篩選 | 預設展示用戶所有訂閱主題的綜合資訊流（依時間逆序）；頂部提供主題標籤篩選器 |
| 內容快取策略 | 新聞元數據快取 24 小時；原文鏈接有效性檢測，失效鏈接標記並隱藏 |
| 實時更新 | 后端定時任務每天 06:00、12:00、18:00 更新新聞；支持手動刷新 |

### 每日 AI 新聞看板 - 實現步驟

#### 1. 新聞聚合與篩選邏輯

- **API 端點**：`GET /api/news/feed` 支援 `topics` 陣列篩選與 `cursor` 無限滾動分頁。
- **Redis 快取架構**：按單一主題建立「Redis 排行榜 (Sorted Set)」。新聞入庫時，以發佈時間戳作 Score 寫入 (`ZADD news:topic:{topic} {timestamp} {article_id}`)。新聞元數據獨立存於 Redis Hash。
- **動態聯集聚合**：
  1. **確認範圍**：取得該次請求的主題列表。
  2. **動態聯集**：利用 `ZUNIONSTORE` 將多個主題的 Sorted Set 瞬間合併成專屬於該次請求的「臨時 Set」(TTL: 60秒)。
  3. **時間排序**：對臨時 Set 執行 `ZREVRANGE` 進行分頁，精準取出 `article_id`。
  4. **批量取值**：透過 Pipeline 批量向 Hash 取回文章詳細內容並回傳。
- **進階優化**：利用 `ZREMRANGEBYRANK` 將各主題在記憶體內的資料量限制為最新 1000 筆，防膨脹；未來可彈性調整 Score 演算法（如加入熱度加權）。

#### 2. 新聞緩存與有效性檢測

- **DB 表結構擴充**：在 `news_articles` 表加入 `cached_until` (快取到期日) 與 `link_valid` (連結有效性) 欄位，並建立索引。
  ```sql
  CREATE TABLE news_articles (
     article_id UUID PRIMARY KEY,               -- 文章的唯一識別碼（主鍵）
     source_url VARCHAR(2048) NOT NULL UNIQUE,  -- 新聞的原始來源網址（設定為唯一值，防止重複抓取）
     title VARCHAR(500) NOT NULL,               -- 新聞標題
     content TEXT,                              -- 新聞完整內文
     summary TEXT,                              -- 新聞重點摘要
     published_at DATETIME,                     -- 新聞在來源網站的實際發佈時間
     fetched_at DATETIME,                       -- 系統後端爬蟲抓取到該篇新聞的時間
     cached_until DATETIME,                     -- 快取預計過期時間（通常設定為抓取後的 24 小時）
     link_valid BOOLEAN DEFAULT true,           -- 原始連結是否仍有效的狀態（預設為有效 true）
     categories JSON,                           -- 該篇新聞所屬的主題分類（例如：["語言模型", "電腦視覺"]）
     created_at DATETIME,                       -- 此筆資料寫入資料庫的建立時間
     updated_at DATETIME,                       -- 此筆資料最後一次被修改的更新時間
  )
  ```
- **背景定時探測**：設置每日任務（如晚上 9 點），針對標記為有效的文章發送網頁標頭狀態請求。

#### 3. 實時更新與爬蟲策略

- **定時批量更新**：排定每日 06:00、12:00、18:00 自動執行背景爬蟲任務，確保早中晚尖峰時段前備妥最新內容。
- **手動即時刷新**：使用者介面提供「重新整理」按鈕允許主動觸發。
- **增量爬取機制**：資料抓取程式啟動時以**時間窗口**為單位（預設 `crawl_window_hours = 6`），抓取最近 6 小時內發布的文章，而非固定筆數。新聞少的日子自然只抓幾篇，爆發日（如重大模型發布）也能完整涵蓋，單一主題單次上限為 `crawl_max_per_topic = 50` 篇防止 API 費用失控。重複 URL 由 `source_url UNIQUE` 約束在資料庫層自動去重。

## 新聞摘要

| 配套項目 | 實現細節 |
|---------|-------|
| 摘要生成 | 透過 LLM 將長文章濃縮為 100~200 字精簡摘要；保留核心觀點與技術細節 |
| 摘要快取 | 相同文章的摘要快取 30 天；重複請求直接返回緩存結果 |

### 新聞摘要 - 實現步驟

#### 1. 摘要生成流程

- 創建 `core/summary/` 目錄存放摘要生成模組
- 實現 `NewsSummarizer` 類，集成 LLM 摘要功能
  ```
  class NewsSummarizer:
    - extract_key_points(article_content) → 提取核心要點
    - generate_summary(key_points, word_limit=150) → 生成 LLM 摘要
    - validate_summary_length(summary) → 驗證字數在 100-200 字內
    - get_summary(article_id) → 先查快取，不存在則生成並存儲
  ```
- **LLM Prompt 設計**：
  ```
  系統提示：你是一位 AI 技術編輯。請用 100-200 字的中文摘要總結以下文章的核心內容。
  要求：1. 突出技術創新點；2. 保留作者主要結論；3. 用簡潔的專業術語；4. 避免冗餘。
  文章內容：{article_content}
  摘要：
  ```

#### 2. 摘要緩存與版本管理
- **DB 表結構**：
  ```sql
  CREATE TABLE article_summaries (
    summary_id UUID PRIMARY KEY,    -- 摘要的唯一識別碼（主鍵）
    article_id UUID NOT NULL UNIQUE, -- 對應的新聞文章識別碼（唯一，一篇文章只對應一筆摘要）
    summary_content TEXT NOT NULL,  -- 由語言模型 (LLM) 生成的摘要完整內容
    word_count INT,                 -- 摘要的總字數（用於快速驗證是否符合 100-200 字的長度限制）
    llm_model VARCHAR(50),          -- 產生摘要所使用的語言模型名稱（例如：GPT-4, Gemini 等）
    generation_time_ms INT,         -- 產生摘要所耗費的時間（單位：毫秒），可用於監控呼叫模型的效能
    cached_until DATETIME,          -- 摘要快取的有效期限（通常設定為生成後的 30 天）
    created_at DATETIME,            -- 此筆摘要記錄寫入資料庫的建立時間
    updated_at DATETIME,            -- 此筆摘要記錄最後一次被修改的更新時間
  )
  ```
- 同一篇文章的摘要在 30 天內不重新生成
- 若文章內容更新，自動刷新相關摘要

## 語音檔生成

| 配套項目 | 實現細節 |
|---------|----------|
| 音檔快取 | 相同新聞的音檔首次生成後快取 30 天；相同新聞不重複生成 |
| 播放控制 | 支持調速（0.75x ~ 1.5x）、暫停記憶位置 |

#### 成本控制

- 監控 TTS API 費用（Google Cloud Speech 等）；設置每用戶月度配額
- 高频播放文章優先用本地合成引擎

### 語音檔生成 - 實現步驟

#### 1. TTS 生成流程與架構

- **API 端點**：`POST /api/news/{article_id}/audio` 請求生成或獲取音檔。
- **生成邏輯**：
  - 先查詢 `audio_caches` 表確認是否已有快取。
  - 若無，提取新聞的 `summary`（優先）或 `content` 作為文本來源。
  - **引擎選擇策略**：判斷文章熱度或排程，優先使用本地開源 TTS 引擎（如 VITS 或 Edge-TTS）免費生成高頻文章；一般文章則調用付費 API（如 Google Cloud TTS 或 OpenAI TTS）。
  - 生成過程採異步處理，前端顯示「生成中...」，完成後透過 WebSocket 或長輪詢通知前端。

#### 2. 音檔存儲與快取

- **DB 表結構**：
  ```sql
  CREATE TABLE audio_caches (
    audio_id         VARCHAR(36) PRIMARY KEY,
    article_id       VARCHAR(36) NOT NULL UNIQUE,
    file_url         VARCHAR(1024) NOT NULL,
    duration_seconds INT,
    tts_engine       VARCHAR(50),    -- 記錄使用的引擎
    file_size_bytes  INT,
    cached_until     DATETIME NOT NULL, -- 快取過期時間（30天後）
    created_at       DATETIME NOT NULL,
    FOREIGN KEY (article_id) REFERENCES news_articles(article_id)
  );

  CREATE INDEX idx_audio_article_id   ON audio_caches(article_id);
  CREATE INDEX idx_audio_cached_until ON audio_caches(cached_until);
  ```
- **存儲位置**：生成的音頻文件（如 `.mp3`）上傳至 AWS S3 或本地靜態對象存儲，資料庫僅存儲 `file_url`。
- **過期清理**：每日定時任務掃描 `cached_until < NOW()` 的記錄，從對象存儲刪除實體檔案並清理 DB。

#### 3. 播放控制與串流

- **串流傳輸**：提供檔案的 CDN 或後端需支援 HTTP `Range` requests，讓前端 `<audio>` 標籤可以平滑緩衝與自由拖曳進度條（Seek）。
- **暫停記憶**：
  - 前端播放時，每隔 5 秒向後端發送當前進度：`PUT /api/user/playback-progress`。
  - 進度數據輕量化存於 Redis Hash `user:{user_id}:playback`，用戶重啟 APP 時從 Redis 讀取上次聽到的秒數。
- **調速實現**：純前端實作，透過修改 HTML5 `<audio>` 元素的 `playbackRate` 屬性達成（支援 0.75x ~ 1.5x 等），無須後端重新生成不同語速的檔案。

#### 4. 成本與配額管理

- **配額計數**：在 Redis 或 DB 中維護用戶當月的 `tts_chars_used`（已消耗字數）。
- **超額處理**：調用付費 TTS API 前檢查配額，超過月度上限時，提示用戶升級方案或強制降級使用免費的本地 TTS 引擎。
- **成本監控**：後端記錄每一筆付費 API 的字數與估算成本，設定每日消費上限（如達到每日 $10 發出告警給運維）。

## 討論版與留言板

### 所需技術工具

- **資料庫**：SQLite - 強大的關聯查詢能力，支援遍歷樹狀結構的遞迴查詢
- **後端框架**：FastAPI - 現代化、非同步處理速度快，且原生支援 WebSocket
- **ORM 映射**：SQLAlchemy - 使用 Python 物件操控資料庫的技術，簡化複雜的 SQL 語法，方便管理留言的層級關係（`comments` 表以 `parent_comment_id` 自我關聯實現樹狀結構）
- **即時通訊**：Socket.io - 封裝了 WebSocket，能處理斷線重連與廣播機能
- **內容過濾**：OpenAI Moderation API - 比純關鍵字過濾更精準，能辨識惡意評論的語意

### 系統功能運行步驟

#### 1. 請求接收與身分安全檢查

系統在此階段接收使用者意圖，並確保通訊安全。

- **API 接入**：後端 FastAPI 接收 RESTful 請求（如 POST /api/v1/comments），並將傳入的 JSON 內容暫存於 new_comment_data 變數中
- **身分驗證**：系統根據 user_id 確認使用者權限，並進行基礎安全掃描，防止 SQL Injection 或 XSS 攻擊

#### 2. AI 內容自動審核

在正式儲存資料前，系統會先過濾不當內容。

- **違規偵測**：系統呼叫 OpenAI Moderation API 進行語意分析
- **評分與判定**：系統計算 moderation_score（毒性分數）並將審核狀態記錄於 is_moderated
- **內容處理**：若內容違規（如仇恨言論），系統立即攔截請求並回傳錯誤訊息，不進入資料庫寫入程序

## 社群平台資訊推送

### 所需技術工具

- **訊息佇列**：Redis - 作為暫存任務的「轉運站」，速度極快，能有效解耦主程式與通知系統
- **任務執行器**：Celery (Python) - 專門從 Redis 抓取任務並在背景執行發送動作，不佔用前端反應時間
- **Email 發送服務**：SendGrid / Gmail SMTP - 提供穩定的郵件發送 API，避免郵件被判定為垃圾信件
- **Discord Bot API**：discord.py - 透過 Bot Token 向指定用戶發送 DM（私訊），需儲存用戶的 `discord_user_id`
- **Facebook Messenger API**：Meta Graph API - 向用戶的 Messenger 推送訊息，需儲存用戶的 PSID（Page-Scoped ID）
- **LINE Messaging API**：LINE Messaging API - 向用戶推送訊息，需儲存用戶的 `line_user_id`

### 系統功能運行步驟

#### 1. 觸發事件與任務封裝

當使用者在系統中觸發特定動作（如獲得回覆、AI 摘要完成或收藏提醒）時，系統會啟動通知流程

- **使用者動作**：使用者在討論區獲得回覆，或 Agent 完成了新聞摘要
- **系統反應**：後端辨識到需要通知使用者，將通知對象、內容與管道封裝成 notification_payload

#### 2. 進入訊息佇列 (Queueing)

- **任務派發**：主程式不會直接呼叫發信函數，而是將任務丟入 Redis 佇列中
- **無感回傳**：此時使用者畫面上會立即顯示「處理完成」或更新內容，完全感覺不到後台正在準備發信

#### 3. Worker 背景執行 (Worker Processing)

- **任務提取**：Celery Worker 偵測到 Redis 中有新任務，主動將其提取出來
- **條件判定**：系統檢查使用者的偏好設定（例如：該用戶設定只接收 LINE 通知，不接收 Email）

#### 4. 第三方 API 調用與發送

- **執行發送**：根據任務類型，呼叫 send_email_notification 或 push_line_msg
- **異常處理**：若第三方 API 暫時失效（如 LINE 伺服器沒回應），Worker 會根據失敗重試機制，在幾分鐘後自動重新嘗試，確保訊息不漏接

#### 5. 使用者接收通知

- **端點送達**：使用者在熟悉的介面（LINE 視窗或電子郵件收件夾）獲取部分消息或摘要連結
- **回流系統**：使用者點擊通知內的連結，引導回到討論版參與互動，完成社群功能循環

## 典藏

### 所需技術工具

- **RAG 框架**：LangChain / LlamaIndex - 用於將新聞原文與語言模型（LLM）結合，自動化處理摘要生成與內容修改的流程
- **向量儲存**：ChromaDB - 本專案使用的向量資料庫，存放典藏文章向量，支援語意搜尋
- **關聯式資料庫**：SQLite（開發）/ PostgreSQL（正式）- 典藏記錄存於 `collections` 表，到期與容量控制皆在此層處理
- **後端框架**：FastAPI - 負責接收收藏請求並串接 AI 處理流程
- **排程系統**：APScheduler - 每日凌晨執行 `cleanup_job()`，掃描 `collection_date` 超過 30 天的記錄並批次刪除

### 系統功能運行步驟

#### 1. 儲存前哨站：容量檢查與自動清理

在資料進入資料庫前，必須先確保符合「50 則」的限制。

- **觸發檢查**：當使用者點擊「收藏」時，系統觸發 check_limit()
- **計數查詢**：執行 SELECT COUNT(*) 查詢該用戶的收藏總數
- **數量判定**：若數量 < 50 則直接進入下一階段；若數量 >= 50 則執行 delete_oldest()，移除時間戳記最早的內容

#### 2. 核心轉化：新聞向量化與摘要生成

這是最耗運算資源的一步，將非結構化的新聞轉為機器可讀的向量。

- **提取與生成**：系統擷取新聞全文，執行 generate_summary_task(raw_content)
- **AI 回傳**：AI Agent 回傳「改寫摘要」與「核心關鍵字」
- **向量轉化**：系統同步執行 bge_m3_encode(summary)，將摘要轉化為一段高維度向量值

#### 3. 寫入與索引

將處理好的數據存入適合的資料庫，以便日後查詢。

- **傳統資料庫**：系統執行 db.insert_text_data()，將新聞 ID、AI 摘要、收藏時間存入
- **向量資料庫**：系統執行 vector_db.upsert_vector()，將步驟二產出的向量值連同 Metadata（新聞 ID）存入

#### 4. 日後查詢與定時維護：RAG 檢索與自動過期

當使用者日後提問時，系統如何從這 50 則中找出答案。

- **查詢檢索**：使用者輸入問題，系統執行 similarity.search(query)，從 50 則收藏中找出關聯性最高的內容
- **LLM 生成**：將找出的內容餵給 LLM，產出有根據的回答
- **定時清理**：系統後台的 APScheduler 每天凌晨執行 `cleanup_job()`，掃描 `collection_date < NOW() - 30 天` 的記錄，同步刪除 `collections` 表資料與 ChromaDB 中對應的向量。

#### 5. 使用者手動刪除

- **API 端點**：`DELETE /api/collections/{collection_id}`，驗證 `user_id` 確認為本人操作。
- **雙層刪除**：先從 ChromaDB 刪除對應的 `vector_id` 向量，再刪除 `collections` 表該筆記錄，確保向量庫與關聯式資料庫不殘留孤立資料。
